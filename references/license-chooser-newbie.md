# License 选择小白指南

第一次开源最容易卡在「选哪个 license」。这里给一句话决策。

## TL;DR

| 你是 / 你怕 | 选这个 |
|---|---|
| **第一次开源，不知道选啥** | **MIT**（默认） |
| 怕大公司用了你的代码再告你专利侵权 | Apache 2.0 |
| 希望别人改了你的代码也必须开源 | GPL 3.0 |
| 完全不在乎别人怎么用，啥都行 | Unlicense / CC0 |

**90% 第一次开源的人选 MIT 不会错**。

## 一句话理由

### MIT（默认推荐）

> 最宽松、最常见。别人能随便用、改、商业化你的代码，**唯一要求是保留你的版权声明**。

**适合**：
- 个人项目 / 副业项目
- 小工具 / 脚本 / 文档
- 希望被广泛使用、不在乎商业化的项目

**不适合**：
- 怕被大公司白嫖然后专利反告（→ Apache 2.0）
- 希望强制别人也开源（→ GPL）

### Apache 2.0

> MIT + 防专利诉讼。别人用了你的代码再起诉你专利侵权 → 他自动失去使用你代码的权利。

**适合**：
- 企业可能用的项目
- 涉及可能有专利的领域（机器学习、加密、算法等）
- 大型开源项目（Kubernetes / TensorFlow 等都用这个）

**不适合**：
- 你完全不在乎专利问题（→ MIT 更短更简单）

### GPL 3.0

> 别人改了你的代码必须也开源（"传染性"）。

**适合**：
- 你希望强制让所有衍生项目也开源
- 你的项目是"反白嫖"立场（如 Linux 内核 / WordPress / Git）

**不适合**：
- 你希望被企业广泛使用（很多公司禁止用 GPL 代码，因为会传染他们的私有项目）
- 不确定 → 选 MIT 比 GPL 安全

### Unlicense / CC0

> 完全放进公有领域。别人怎么用都行，连保留作者名都不要求。

**适合**：
- 一次性脚本 / 小代码片段
- 不想给别人加任何使用门槛

**不适合**：
- 几乎所有"真项目"（连保留作者名都不要求会让人怀疑可信度）

## 决策树（30 秒）

```
你的项目能被大公司商业化用吗？
├─ 不在乎     → MIT
├─ 怕专利问题 → Apache 2.0
└─ 不许，必须也开源 → GPL 3.0
```

## 不要做的事

- **不要不加 License**。没 License 默认是"保留所有权利"，别人法律上不敢碰你的代码，等于发了也没人敢用
- **不要自己写 License 文本**。法律细节坑很多。从 https://choosealicense.com/ 或本仓 `references/license-templates/` 直接复制
- **不要在 LICENSE 里改版权年份和姓名以外的任何字**。改了一个字法律上就不再是标准 license

## 你必须填的两个字段

不管选哪个 license，LICENSE 文件里有两个东西必须替换成你自己的：

1. **版权年份**：通常是 `<year>` 占位，改成今年（如 `2026`）
2. **版权所有人**：通常是 `<copyright holders>`，改成你的名字 / GitHub username / 公司名

`scripts/generate_files.py` 会自动替换这两个。

## 后悔了能改吗？

**改宽不能改严**。MIT 改成 GPL → 法律上等于你单方面收紧，但**别人之前以 MIT 拿到的版本不受影响**。所以选的时候宁可保守也别选错。

MIT 是最保守的（最宽松、最广接受），所以选不准就选 MIT。

## 第三方代码 / Fork 注意

如果你的项目是 **fork** 别人的，或者包含 **第三方代码（vendor/ 等）**，你不能完全自己定 license：

- Fork 的：必须沿用 upstream 的 license（不能 fork GPL 项目然后改成 MIT 重发）
- 第三方代码：**保留原 LICENSE 文件**不要删，README 加 Acknowledgements

`scripts/scan_project.py` 会自动检测这两种情况。
